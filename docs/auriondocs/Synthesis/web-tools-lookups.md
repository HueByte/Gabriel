# Web tooling and external lookups

> Integration with web search tools, fetchers, and tool bridges used by the AI stack.

This guide explains the small set of web-search implementations and the tool-bridging machinery that lets a text-only chat provider emit structured tool calls for the agent runtime. Read these notes to understand what each component actually does (parsing behavior, error semantics, and streaming/event shapes) and how they hand off work to each other at runtime.

## DuckDuckGoWebSearch
Provides a DuckDuckGo-based web search implementation.

[DuckDuckGoWebSearch](../Code/src/api/Gabriel.Infrastructure/Tools/Web/DuckDuckGoWebSearch.cs.md) is an [IWebSearch](../Code/src/api/Gabriel.Infrastructure/Tools/Web/DuckDuckGoWebSearch.cs.md) implementation that scrapes DuckDuckGo's public HTML endpoints rather than calling a formal API. Concretely, it prefers the richer html/ endpoint and falls back to the lite/ endpoint when parsing fails or an anomaly/throttling page is observed; it exposes methods like SearchAsync, FetchAsync, ParseHtmlEndpoint/ParseLiteEndpoint and utility helpers (BuildRequest, EnsureSessionAsync, UnwrapRedirect, ResetSession) to manage requests and parsing. The class keeps lightweight session state (a CookieContainer warmed via EnsureSessionAsync and a chosen browser User-Agent) and uses a SemaphoreSlim to serialize first-use session warmup; it logs when it detects anomaly pages and uses regex-driven parsing (HtmlResultBlockRegex, HtmlSnippetRegex, LiteResultLinkRegex, etc.), a brittle but keyless approach. Relationships: No explicit relationships within this topic were provided; the doc block identifies it only as an IWebSearch implementation that can be used wherever IWebSearch is consumed.

## BraveWebSearch
Provides Brave-based web search integration.

[BraveWebSearch](../Code/src/api/Gabriel.Infrastructure/Tools/Web/BraveWebSearch.cs.md) is a straightforward [IWebSearch](../Code/src/api/Gabriel.Infrastructure/Tools/Web/BraveWebSearch.cs.md) adapter over Brave Search's /search endpoint that issues a GET and maps the API response into domain [WebSearchResult] objects. It expects a DI-configured named HttpClient (the docs refer to a "BraveSearch" client) and throws an InvalidOperationException if the configured API key is missing, failing fast to surface misconfiguration. The class enforces a hard result limit by clamping caller-supplied limits into the 1–10 range, and it tolerantly substitutes empty strings for missing title/URL fields to keep the consumer surface stable. Relationships: No explicit relationships within this topic were provided; it implements IWebSearch and is intended to be used where an IWebSearch is required.

## GabrielToolBridge
Bridges chat provider with tools and exposes streaming interface.

[GabrielToolBridge](../Code/src/api/Gabriel.Engine/Providers/ToolBridge/GabrielToolBridge.cs.md) is an [IChatProvider](../Code/src/api/Gabriel.Engine/Providers/ToolBridge/GabrielToolBridge.cs.md) decorator that adapts a tool-incapable inner provider to the event-and-wire-format the agent runtime expects. It injects a system message documenting the available tools (via InjectToolDocs/BuildToolDocs/EscapeJsonString), calls the inner provider with an empty tools list, streams text from the inner provider while watching for embedded <tool_call> markers, and parses buffered tail text into structured tool-call events in StreamAsync; helper methods like SerializeCallInline and AppendFixupMessage assist with producing or repairing calls. Importantly, the bridge does not execute tools itself — it emits events (including ToolCallReady) for downstream execution — and it only streams text for the first parse attempt (subsequent parse retries produce only finalized tool-call events to avoid "unsending" text). Relationships: depends on and uses [ToolCallBlockParser](../Code/src/api/Gabriel.Engine/Providers/ToolBridge/ToolCallBlockParser.cs.md) to turn embedded <tool_call> blocks into structured ParsedToolCall objects; it also emits events shaped as [ChatProviderEvent](../Code/src/api/Gabriel.Engine/Providers/ChatProviderEvent.cs.md) instances.

## ChatProviderEvent
`ChatProviderEvent` collaborates directly with `GabrielToolBridge` and other members of this topic (6 dependency links).

[ChatProviderEvent](../Code/src/api/Gabriel.Engine/Providers/ChatProviderEvent.cs.md) is the abstract base record for the streaming events that an IChatProvider produces from StreamAsync. The concrete derived records enumerated in the file include terminal markers like FinishEvent (with a FinishReason), delta events such as TextDeltaEvent and ReasoningDeltaEvent, and a ToolCallReadyEvent that represents a fully assembled tool call. Providers buffer partial JSON/tool-call fragments internally and only emit a ToolCallReady event when the tool invocation is complete, so consumers always get cohesive, fully formed tool-call events rather than fragmented JSON pieces. Relationship: used by [GabrielToolBridge](../Code/src/api/Gabriel.Engine/Providers/ToolBridge/GabrielToolBridge.cs.md) as the event envelope it streams to callers.

## ToolCallBlockParser
`ToolCallBlockParser` collaborates directly with `GabrielToolBridge` and other members of this topic (5 dependency links).

[ToolCallBlockParser](../Code/src/api/Gabriel.Engine/Providers/ToolBridge/ToolCallBlockParser.cs.md) scans a text stream for <tool_call>...</tool_call> blocks and converts each into a typed ParsedToolCall, assigning a synthetic id (emu_<12-char-guid>_<index>) to aid per-turn correlation. The parser enforces that each block contains a JSON object with a string "name" and an optional "arguments" JSON object, preserves the raw arguments JSON (substituting "{}" when arguments are absent), and throws a ToolCallParseException for unterminated blocks or invalid JSON to drive retry logic. It deliberately stops short of interpreting tool names or arguments beyond validation/serialization — execution and mapping to actual tool implementations are left to downstream components. Relationship: invoked by [GabrielToolBridge](../Code/src/api/Gabriel.Engine/Providers/ToolBridge/GabrielToolBridge.cs.md) to turn inline tool-call markers detected in streamed text into [ParsedToolCall](../Code/src/api/Gabriel.Engine/Providers/ToolBridge/ToolCallBlockParser.cs.md) instances and to surface parse errors via ToolCallParseException.

How the pieces fit

At runtime the bridge pattern is: a chat provider that cannot natively call tools is wrapped by [GabrielToolBridge](../Code/src/api/Gabriel.Engine/Providers/ToolBridge/GabrielToolBridge.cs.md), which injects tool documentation into the prompt, streams the provider's text, and detects embedded <tool_call> markers. The bridge buffers and, on parse completion, emits strongly typed [ChatProviderEvent](../Code/src/api/Gabriel.Engine/Providers/ChatProviderEvent.cs.md) events (including ToolCallReady) after turning text fragments into [ParsedToolCall] objects via [ToolCallBlockParser](../Code/src/api/Gabriel.Engine/Providers/ToolBridge/ToolCallBlockParser.cs.md); execution of those parsed calls is intentionally performed by the server-side tool pipeline rather than the bridge. Separately, the two IWebSearch implementations ([DuckDuckGoWebSearch](../Code/src/api/Gabriel.Infrastructure/Tools/Web/DuckDuckGoWebSearch.cs.md) and [BraveWebSearch](../Code/src/api/Gabriel.Infrastructure/Tools/Web/BraveWebSearch.cs.md)) provide concrete web-search capabilities that can be bound into the tool pipeline the bridge delegates to; DuckDuckGo offers a keyless, HTML-scraping fallback with session-warming and anomaly detection, while BraveWebSearch calls Brave's API and enforces API-key presence and result limits.
