DuckDuckGoWebSearch_overview is a free, API-key-free search provider implementation that uses DuckDuckGo as its backend. It tries two endpoints in this order: html.duckduckgo.com/html/ as the primary source of rich HTML results, and, if no parseable results are produced, lite.duckduckgo.com/lite/ as a lightweight fallback. Both endpoints receive the same form payload, with the lite endpoint designed for accessibility and low-bandwidth scenarios and being less aggressively flagged, making it a useful safety net when the HTML path returns content that cannot be parsed. The class relies on dependency-injected IHttpClientFactory and ILogger, maintains simple session state including a per-session User-Agent and a single-session warmup flag, and uses a semaphore to deduplicate concurrent first-use warmups. When DuckDuckGo returns an anomaly or CAPTCHA-like page, diagnostic details are logged to keep failures debuggable rather than silent. This approach has no quota or API key requirements, but HTML shape can change and the parsing is regex-driven rather than DOM-based, so reliability is lower than a paid API. For production-scale reliability, a paid backend (e.g., Brave with an API key) is generally preferable.

## Remarks
DuckDuckGoWebSearch_overview encapsulates the complexity of talking to DuckDuckGo behind a single IWebSearch interface. It provides resilience by falling back from rich HTML results to a minimal lite page and centralizes session handling (Cookies and User-Agent) so callers see a stable, uniform surface. It fits alongside its collaborators (IWebSearch, IHttpClientFactory, ILogger) by isolating network and parsing concerns from consumer code, enabling swapping the underlying backend without changing the consumer surface.

## Notes
- HTML parsing is regex-driven and brittle: changes to DuckDuckGo’s HTML can break parsing, and parse failures yield zero results rather than exceptions.
- The lite endpoint is a safety net: while it is less likely to be flagged and consumes fewer resources, it may return sparser data or be less informative than the rich HTML path.
- Session state is per-instance (UA and warmup flag); the first-use warmup is deduplicated via a semaphore, so ephemeral or short-lived instances may not share state as expected. Long-lived, DI-managed instances are preferred to preserve session behavior.
